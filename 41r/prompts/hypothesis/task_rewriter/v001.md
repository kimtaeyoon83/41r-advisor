# Persona × Sub-Question → 1인칭 Task

특정 페르소나가 특정 sub-question을 검증하기 위해 사이트에서 할 **행동 지시**를 작성한다.
페르소나의 soul을 반영해 **그 사람의 관점**으로 써야 한다.

## 입력
- `persona_soul`: 페르소나의 soul YAML + 서술 (요약형)
- `persona_traits`: 5축 trait 값
- `sub_question`: {id, text, observable_signals}
- `url`: 사이트 URL
- `hypothesis`: 원 가설 (맥락 보존용)

## 원칙
1. **1인칭**. "나는 ...한다" 형태. agent_loop의 plan 프롬프트가 이걸 그대로 받음.
2. **행동 명시**. "판단한다" 아니고 "3초 본 뒤 닫을지 결정한다".
3. **페르소나 특성 반영**. 같은 sub-question도 impulsive는 "즉시", cautious는 "신호 찾기 후".
4. **짧게**. 2~3문장. 너무 길면 agent가 혼란.
5. **결과 형태 지정**. "이탈하면 drop_point를, 머물면 key_behaviors를 기록" 같은 힌트.

## 출력 (JSON만)
```json
{
  "task": "1인칭 task 문장 (2~3줄)",
  "expected_outcome_hint": "task_complete | abandoned | partial 중 예상되는 것",
  "focus_fields": ["drop_point", "frustration_points"]
}
```

## 예시

**입력**:
- persona: p_impulsive (20대 남, 즉흥적, 시각 의존 높음)
- sub_question: "3초 첫인상에서 이탈하는가"
- hypothesis: "MZ에게 먹히나"
- url: toss.im

**출력**:
```json
{
  "task": "나는 친구가 추천한 금융앱을 처음 만났다. 첫 3초 동안 눈길 끄는 게 없으면 바로 뒤로 간다. 색감, 첫 카피, 버튼 중 하나라도 '쿨하다'는 느낌이 없으면 이탈한다.",
  "expected_outcome_hint": "abandoned",
  "focus_fields": ["drop_point", "frustration_points"]
}
```

**다른 예시**:
- persona: p_senior, 같은 sub_question →
```json
{
  "task": "나는 TV에서 본 적 있는 금융앱인지 확인하러 들어왔다. 익숙하지 않은 디자인이나 모르는 용어가 처음에 보이면 당황해서 뒤로 간다. 큰 글씨, 명확한 버튼 이름이 없으면 멈춘다.",
  "expected_outcome_hint": "abandoned",
  "focus_fields": ["drop_point", "frustration_points"]
}
```

같은 sub_question이지만 persona별로 **이탈 기준**이 다르게 쓰여짐.
