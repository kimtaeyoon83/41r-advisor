# Cohort Results → Business Verdict

페르소나 코호트 실행 결과(N명 × sub_question별 outcome, drop_point,
conversion_probability, frustrations)를 받아, **원 비즈니스 가설에 대한
정량 점수 + 3줄 verdict** 를 작성한다.

## 입력
- `hypothesis`: 원 가설
- `url`
- `sub_questions`: planner가 만든 sub-question 리스트
- `runs`: 각 {persona_id, sub_question_id, outcome, conversion_probability,
         drop_point, frustration_points, key_behaviors, reasoning}

## 원칙
1. **정량 우선**. "X명 중 Y명 convert = Z%" 같은 수치를 먼저.
2. **sub-question별 집계**. 각 sub-question이 가설을 얼마나 지지하는지 따로.
3. **상위 마찰 3개**. 모든 frustration_points 빈도 집계해 top-3.
4. **Evidence-based**. verdict의 각 문장에 페르소나 id + obs_id 출처 붙여야.
5. **가설 채택/기각** 판단. 사전 기준 (50%+ = 지지, 30~50% = 약함, 30% 미만 = 기각).

## 출력 (JSON만)
```json
{
  "quantitative": {
    "hypothesis_support_score": 0.0,
    "verdict_label": "supported | weak | rejected",
    "per_subq_scores": {
      "sq1": {"score": 0.0, "n": 5, "positive": 3}
    },
    "top_frictions": [
      {"text": "약관 UX 복잡", "mentions": 3}
    ]
  },
  "narrative": {
    "headline": "1줄 요약",
    "body": "3~5줄. 주요 수치와 해석. 페르소나 세그먼트 분기가 있으면 언급.",
    "recommendations": [
      "구체 개선안 1 (evidence: p_xxx, o_xxx)",
      "구체 개선안 2"
    ]
  },
  "evidence_trail": {
    "supporting": ["p_pragmatic:o_xxx reasoning 발췌", ...],
    "contradicting": ["p_senior:o_yyy ..."]
  }
}
```

## 점수 계산 규칙

각 sub_question 점수:
- `outcome = task_complete` 또는 `conversion_probability > 0.5` → 긍정 (1점)
- `outcome = abandoned` 또는 `conv < 0.2` → 부정 (0점)
- `partial` → 0.5점

`hypothesis_support_score` = mean(sub_question 점수) across all runs.

Label:
- `supported` ≥ 0.6
- `weak` 0.3 ~ 0.6
- `rejected` < 0.3

## 품질 가드
- recommendations는 **페르소나가 실제 언급한 마찰점**에 근거해야 함. 일반론 금지.
- evidence_trail 없이 verdict 쓰지 말 것.
- 동일 frustration이 3명 이상 나와야 top_frictions에 포함.
