# Hypothesis → Observable Sub-Questions

비즈니스 가설 (한 줄) 을 받아서, **페르소나 행동으로 관찰 가능한**
sub-question 3~5개로 분해한다. 각 sub-question에는 어떤 성향의
페르소나가 검증에 적합한지 힌트를 붙인다.

## 입력
- `hypothesis`: 한 줄 가설 (예: "이 사이트가 MZ에게 어필할까?")
- `url`: 대상 URL
- `context`: (선택) 사이트 카테고리 힌트

## 원칙
1. **주관 제거**. "어필할까"가 아니라 "3초 내 이탈률"처럼 **측정 가능**한 형태로.
2. **행동 신호**. outcome / conversion_probability / drop_point / time 같은
   agent_loop이 자동으로 채우는 필드와 연결되게.
3. **3~5개 이내**. 너무 많으면 노이즈.
4. **page_state 접근 가능**. 페르소나가 "페이지를 본 뒤" 판단할 수 있는 것만.

## 출력 (JSON만, 코드펜스 안에)
```json
{
  "sub_questions": [
    {
      "id": "sq1",
      "text": "첫 화면 3초 노출 후 이탈 의향이 생기는가",
      "observable_signals": ["outcome=abandoned", "drop_point=hero section"],
      "target_traits": {
        "impulsiveness": ">0.7",
        "visual_dependency": ">0.5"
      },
      "rationale": "MZ 어필의 첫 관문은 초단위 시각 즉시성"
    },
    ...
  ],
  "aggregate_plan": "각 sub-question의 페르소나 N명 집계 후, X% 이상이 긍정이면 가설 채택",
  "recommended_cohort_size": 5,
  "recommended_mode": "text"
}
```

## 예시 (참고용)
가설 "MZ에게 먹힐까?" → sub_questions:
- 3초 첫인상 (visual, impulsive)
- 30초 내 가입 의향 (impulsive, low research)
- 공유 의향 발견 (social_proof high)

가설 "신뢰감 있나?" → sub_questions:
- 회사 정보 접근성 (privacy high)
- 구체적 수치·사례 노출 (research_depth high)
- 보안 시그널 탐지 (privacy high, cautious)

가설 "전환이 잘 되나?" → sub_questions:
- CTA 발견 속도 (visual + goal-oriented)
- 결제까지 마찰 수 (price_sens, pragmatic)
- 의심 트리거 (cautious, privacy)
